## Phase 10 Review (carried into Phase 11 prompt)

- **`LocalLanBackend` surface (`com.aikeyboard.app.ai.client.locallan`):** plain `AiClient` impl taking `(baseUrl, apiFormat, apiKey, modelName)` constructor args. `isAvailable()` returns true when `baseUrl` and `modelName` are both non-empty (mirrors `RemoteApiBackend`/`TermuxBridgeBackend` "do you have what's needed to *try*" semantics). `rewrite()` is a `callbackFlow` that branches on `apiFormat` between `streamOllama` (POST `<base>/api/chat`, line-delimited JSON) and `streamOpenAi` (POST `<base>/v1/chat/completions`, SSE) — both use `Ktor.preparePost(...).execute { ... }` with the OkHttp engine, body-as-channel, line-by-line delegation to the format-specific parser. Reuses `TermuxBridgeBackend.httpClient` (60s socket, 5s connect — exposed as a `companion`-default-public field per Phase 6) — avoids a second `HttpClient` instance with a duplicated OkHttp engine. Catch order is `TimeoutCancellationException` → `CancellationException` → `Throwable` (mirrors the catch-order fixes from Phase 6's `TermuxBridgeBackend` and Phase 7b's `RemoteApiBackend` incidental fix; the inverse order silently swallows timeouts because `TimeoutCancellationException IS-A CancellationException`). HTTP-level error mapping: 401/403→`AUTH_FAILURE`, 429→`RATE_LIMITED`, 5xx→`NETWORK_FAILURE`, else→`UNKNOWN`. Optional `Authorization: Bearer <key>` header appended only when `apiKey.isNotEmpty()`.

- **Stream parsers (pure JVM, no Ktor / Android):** `OllamaStreamParser.parseLine(line, emit)` decodes `{message:{role,content}, done, error}` chunks; the `error` field is mapped to a static `"ollama-error"` sentinel to prevent leaking server-side error strings (which can include prompt fragments). `OpenAiCompatStreamParser.parseLine(line, emit)` consumes SSE lines: skips blank lines, SSE comments (`:` prefix), non-`data:` lines (`event:`, `id:`), and empty data payloads (heartbeats); emits `Done` on the `[DONE]` sentinel; emits `Done` on terminal `finish_reason` values (`stop`, `length`) but explicitly NOT on intermediate ones (`tool_calls`, `content_filter`) which appear on chunks that aren't the last one. Both parsers wrap JSON decode in `runCatching { ... }.getOrNull()` so malformed input never propagates a `JsonDecodingException` whose message could echo the raw line.

- **`PublicIpValidator` (`com.aikeyboard.app.ai.client.locallan.util`):** pure-JVM host classifier returning one of `LOOPBACK / PRIVATE / LINK_LOCAL / HOSTNAME / PUBLIC / INVALID`. `classifyUrl(url)` extracts the host via `java.net.URI(url).host` then defers to `classifyHost(host)`. IPv4 ranges checked exactly (`10/8`, `172.16/12`, `192.168/16`, `127/8`, `169.254/16`, `0.x` and `>=224` flagged as `INVALID`). IPv6 narrow: `::1` / `[::1]` → `LOOPBACK`, `fe80:` prefix → `LINK_LOCAL`, anything else with a colon → `PUBLIC` (conservative — ULA / GUA classification deferred to Phase 12). `isSafeForCleartext(c)` returns true for everything except `PUBLIC` and `INVALID`, gating the warn-before-save dialog in the edit screen.

- **Storage extensions (`SecureData` + `SecureStorage`):** four new default-valued fields on `SecureData` (back-compat with Phase 1–9b blobs; no schema bump): `localLanBaseUrl: String = ""`, `localLanApiFormat: String? = null`, `localLanApiKey: String = ""`, `localLanModelName: String = ""`. Eight new `SecureStorage` accessors (`get`/`set` for each), all `@Synchronized` matching Phase 6/8 pattern. `getLocalLanApiFormat()` falls back to `LocalLanApiFormat.OLLAMA` on null (default-on-first-launch) or unknown enum value (graceful for future renames) — same `runCatching { LocalLanApiFormat.valueOf(...) }.getOrNull()` pattern as `getSelectedBackendStrategy`.

- **`BackendResolver` extension:** `LOCAL_LAN` branch reads `getLocalLanBaseUrl()` + `getLocalLanModelName()`, returns null if either is empty (consistent with `REMOTE_API`/`TERMUX_BRIDGE` null-on-incomplete-config), else constructs `LocalLanBackend(...)`. Function signature converted from expression body to block body because the new branch needs `if (...) return null` instead of an Elvis early-return — Kotlin rejects naked `return` in `when` expressions inside expression-body functions.

- **Compose surface:** new `LocalLanEditRoute` + `LocalLanEditScreen` under `com.aikeyboard.app.ai.ui.locallan`. `LocalLanEditRoute` is the state hoist (loads existing values from `SecureStorage`, persists on `onSave`). `LocalLanEditScreen` is a vertically-scrolling form with five fields (Base URL, API format radio group, Model name, optional API key with show/hide toggle, Save button). The Save button is disabled while either `baseUrl` or `modelName` is empty post-trim. On save, `PublicIpValidator.classifyUrl(baseUrl.trim())` runs; if `!isSafeForCleartext(...)`, an `AlertDialog` (`ai_settings_locallan_public_ip_*` strings) intercepts with "Save anyway" / Cancel buttons. Cancel button re-uses the existing `ai_settings_persona_cancel` string per Phase 9b's "no unused strings" lesson. `BackendsScreen` gained a new `onOpenLocalLanEdit` parameter and a fourth row (positioned between Remote API providers and Termux Bridge to match `BackendStrategy` enum order: `REMOTE_API → LOCAL_LAN → TERMUX_BRIDGE`). The row carries its own embedded `RadioButton` (selectable iff base+model both non-empty, mirroring the Termux row's pattern), an inline status string (`"<host> · <model>"` when configured, `"<host> · model required"` when partially configured, `"Not configured"` when empty), and routes onClick to the new `BACKENDS_LOCALLAN_EDIT` route. `AiSettingsNavHost` adds the route constant + composable + the new `onOpenLocalLanEdit` callback wiring.

- **Network security config — privacy-posture trade-off:** `base-config` switched from `cleartextTrafficPermitted="false"` to `"true"` so user-configured Local LAN base URLs can reach RFC1918 / mDNS / hostname destinations (Android NSC does not support CIDR ranges or IP-literal wildcards, so we cannot statically allow only RFC1918). The two cloud-provider `<domain-config>` blocks (Anthropic, Gemini) RETAINED with `cleartextTrafficPermitted="false"` as defense-in-depth — they continue to forbid cleartext to those hostnames even though base now permits it. The previous `<domain-config cleartextTrafficPermitted="true">` block for `127.0.0.1` / `localhost` was REMOVED (subsumed by base-config's permissive default; Termux loopback continues to work). The `<base-config>` element carries `tools:ignore="InsecureBaseConfiguration"` since lint flags this as a security warning by design — the suppression is the documented decision, not a baseline-pollution attempt.

- **Privacy compensator (in code):** (1) `LocalLanBackend` is the ONLY backend whose URL is user-controlled — `RemoteApiBackend` HTTPS-only via the retained domain-configs, `TermuxBridgeBackend` hardcoded to `127.0.0.1:8787`. (2) `PublicIpValidator` + the warn-before-save dialog ensures users explicitly acknowledge cleartext to a public IP. - **Deviations from Phase 10 prompt:**
  - **`network_security_config.xml` `base-config` switched from `cleartextTrafficPermitted="false"` to `"true"`.** Required because Android NSC does not support CIDR ranges or IP-literal wildcards, so we cannot statically allow only RFC1918. The privacy compensator is `PublicIpValidator` + the warn-before-save dialog: `LocalLanBackend` is the only backend whose URL is user-controlled, and the UI guards against accidentally saving a public-IP cleartext URL. (This is the deviation the prompt itself flagged in §6 as the documented trade-off.)
  - **`tools:ignore="InsecureBaseConfiguration"` added to the `<base-config>` element.** Lint flags `cleartextTrafficPermitted="true"` on a `base-config` as a security warning. Without the ignore, the warning either becomes a new entry in `lint-baseline.xml` (violating PHASE_REVIEW.md's "no new baseline entries this phase" rule) or surfaces as a non-baseline lint warning (violating Phase 10 DoD §18). Inline suppression is the cleanest path — the trade-off is already documented in this summary's "privacy compensator" section.
  - **`BackendResolver.resolve()` converted from expression body to block body.** The new `LOCAL_LAN` branch needs `if (baseUrl.isEmpty() || model.isEmpty()) return null` to short-circuit, but Kotlin rejects naked `return` inside a `when` expression in an expression-bodied function (`error: Returns are prohibited in functions with expression body`). The existing `?:` Elvis-return idiom doesn't work for the two-condition test. Block body is the minimum-diff path; behavior identical.
  - **`BackendResolverTest.kt` lost the original `localLan_alwaysReturnsNull_phase10Owns` test (it was wrong now)**, replaced with three more-specific tests: `localLan_unconfigured_returnsNull` (empty URL+model), `localLan_baseUrlSetButNoModel_returnsNull` (partial config), and `localLan_fullyConfigured_returnsLocalLanBackend` (full happy path). Net +2 to BackendResolverTest's count (5 → 7).
  - **Five strings prompt-listed but dropped to satisfy lint:** `ai_settings_backends_radio_locallan` (the prompt's §15 last entry) was specified in the prompt but the implementation reuses `ai_settings_backend_locallan_title` for the row's headline (which is the only place a "Local LAN" label is rendered). Dropping it avoids a `UnusedResources` lint failure — Phase 9b's "four dropped strings" precedent. The prompt's other 16 strings are all wired and rendered.
  - **`OpenAiCompatStreamParser` uses `@SerialName("finish_reason")`** on the `OpenAiChoice.finishReason` Kotlin field. The prompt's example wrote `finish_reason` as a Kotlin identifier directly; renaming to camelCase plus a `@SerialName` annotation is idiomatic Kotlin/kotlinx.serialization and produces the same wire format.
  - **`OpenAiCompatStreamParser.parseLine` treats empty `data: ` payloads as no-ops** even though the prompt's example didn't explicitly test for them. Some servers send `data:` (followed by a single space and CRLF) as a heartbeat — without the early-return, `JSON.decodeFromString("")` would throw a `MissingFieldException` whose message includes `<empty>` (privacy-clean, but emit-nothing is the cleaner contract). Test `emptyDataPayload_isNoOp` covers it.

- **Open questions for human reviewer (carry into Phase 11 prompt):**
  1. **NSC permissive base-config — F-Droid metadata flags?** F-Droid's index sometimes warns on `usesCleartextTraffic="true"` / `cleartextTrafficPermitted="true"` and may add an anti-feature label. Phase 12 release-prep needs to verify whether the in-code guardrails (`PublicIpValidator`, the warn-before-save dialog, the `LocalLanBackend`-only user-URL surface) are sufficient justification for an explicit anti-feature listing, or if the permissive base-config triggers a hard block.
  2. **Health check / "Test connection" button.** Phase 3a deferred this for `BackendEditScreen`; Phase 10 does the same for `LocalLanEditScreen`. Phase 12 polish: add a "Test connection" button that fires `GET /api/tags` (Ollama) / `GET /v1/models` (OpenAI-compat) and surfaces a green/red status. Defer unless smoke testing shows users get stuck at "configured but not actually reachable."
  3. **mDNS / `.local` discovery.** `PublicIpValidator` classifies `*.local` as `HOSTNAME` (treated as safe), so users can save a `pi.local` URL without warning. Actual mDNS resolution requires `android.permission.CHANGE_NETWORK_STATE` and explicit NSD setup — out of scope for v1.
  4. **Auto-detection between Ollama and OpenAI-compat formats.** Phase 10 makes the user pick. Auto-detect (`/api/tags` 200 → Ollama, 404 → OpenAI-compat) is feasible but failure-prone across the long tail of self-hosted servers. Defer to Phase 12 if the radio choice produces support friction.
  5. **IPv6 classification beyond loopback.** Phase 10's `PublicIpValidator` treats most IPv6 as `PUBLIC` conservatively. If users run Ollama on IPv6 ULA (`fc00::/7`), they get the warn-before-save dialog. Acceptable v1; Phase 12 can add full IPv6 classification (`fc00::/7` ULA, `fe80::/10` link-local, `2001:db8::/32` documentation, etc.).
  6. **Ollama `keep_alive` parameter.** Ollama's `/api/chat` accepts an optional `keep_alive` field (e.g., `"5m"`). Phase 10 omits it (server uses default). If users see slow first-token latency, Phase 12 can expose this as an advanced field.
  7. **`finish_reason` heuristic on OpenAI-compat parser.** Phase 10 emits `Done` only on `stop` and `length`; `tool_calls` / `content_filter` are intentionally NOT terminal so the stream can continue. If a server emits a `finish_reason` we don't recognize on the actual final chunk, the stream falls through to the loop-end (channel closed) and emits `Done` there — same fallback as Phase 3b's Anthropic SSE path. Worth checking against vLLM / Together / Groq variants in Phase 12 polish.
