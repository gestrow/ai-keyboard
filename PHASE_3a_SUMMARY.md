## Phase 3a Review (carried into Phase 3b prompt)

- **Pinned dependency versions:** `com.google.crypto.tink:tink-android:1.21.0` (released 2026-03-24, latest stable per github.com/tink-crypto/tink-java releases). Ktor `3.4.3` for `io.ktor:ktor-client-okhttp` + `ktor-client-content-negotiation` + `ktor-serialization-kotlinx-json` (released 2026-04-23, latest stable per github.com/ktorio/ktor releases). `kotlinx-serialization-json:1.9.0` already present from Phase 2 — reused. `androidx.security:security-crypto:1.1.0` retained for one-shot legacy migration only, comment in `build.gradle.kts` flags Phase 12 as the removal point.

- **Storage layer:** Single Tink AEAD blob at `context.filesDir/ai_keyboard_secure.bin`, AES-256-GCM, master keyset wrapped by Android Keystore via `AndroidKeysetManager` with master URI `android-keystore://ai_keyboard_master_key`; keyset metadata in `shared_prefs/ai_keyboard_keyset_prefs.xml`. Envelope is a `@Serializable internal data class SecureData(personas, activePersonaId, apiKeys)` extracted to its own file so JVM unit tests can exercise the JSON round-trip without Android Keystore. AAD `"ai_keyboard_secure_v1"` binds ciphertext to this data shape (defense in depth). Ciphertext written via tmp-file + atomic rename (with overwrite fallback).

- **Deviations from Phase 3a prompt:**
  - **`AiClient` is a plain `interface`, not `sealed interface`** as the prompt's example declares. Kotlin sealed types require subtypes in the same package, and the prompt itself places `RemoteApiBackend` in `com.aikeyboard.app.ai.client.remote` (with `LocalLanBackend` + `TermuxBridgeBackend` due in their own subpackages later). Closed-set semantics live on `BackendStrategy` instead. Inline comment in `AiClient.kt` flags the choice for reviewers.
  - **`SecureData` extracted to a separate file (`SecureData.kt`)** rather than nested private inside `SecureStorage`, so the JVM unit tests can exercise the JSON envelope without instantiating `SecureStorage` (which needs Android Keystore). Marked `internal`. Public surface of `SecureStorage` is unchanged from the prompt.
  - **Eager `SecureStorage.getInstance(this)` call in `App.onCreate`** (immediately after `FirstRunDefaults.apply(this)`). Without this, migration only runs when the keyboard binds; we wanted package-replaced broadcasts to migrate too. Migration's existing try/catch absorbs the direct-boot-mode case; the lazy `aead by lazy` means we don't pay Keystore-init cost on every launch.
  - **New `ic_chevron_right.xml` VectorDrawable** added because HeliBoard's `ic_arrow_right.xml` is a `<rotate>` wrapper around `ic_arrow_left`, which Compose's `painterResource` rejects with `IllegalArgumentException: Only VectorDrawables and rasterized asset types are supported`. Used by SettingsHubScreen + BackendsScreen list rows. Same upstream-icon-coupling concern still applies elsewhere (Phase 2 carried it forward already).
  - **`AndroidKeysetManager` (the older API) used instead of the newer `AndroidKeystore` + `TinkProtoKeysetFormat` pattern** that Tink 1.21.0's `examples/android/helloworld` ships. The prompt explicitly named `AndroidKeysetManager`; both APIs are in 1.21.0. The newer pattern would be the right Phase 12 cleanup if Tink ever deprecates `AndroidKeysetManager`.
  - **Anthropic request body adds `accept: text/event-stream` header** beyond what the prompt lists; required by the SSE stream we'll consume in 3b. Same for Gemini. Doesn't change request semantics for the 3a stub.

- **Carried issues:**
  - **2 deprecation warnings on `EncryptedSharedPreferences` + `MasterKey` imports in `SecureStorage.kt`**: intentional, scoped to the migration method. `@Suppress("DEPRECATION")` covers the call sites but not the imports. Resolves when Phase 12 deletes the migration code + the `security-crypto` dependency.
  - **2 deprecation warnings on `LocalLifecycleOwner`** in `PersonaListScreen.kt` and `BackendsScreen.kt`. Same pre-existing issue noted in Phase 2 — fixed by adding `androidx.lifecycle:lifecycle-runtime-compose`. Defer to Phase 12.
  - **Direct-boot-mode migration corner case**: if `App.onCreate` runs while the device is locked (e.g., DIRECTBOOTAWARE startup), Tink's `AndroidKeystore` is unavailable and migration's try/catch swallows the failure. The singleton then caches an empty `SecureData`, and subsequent `getPersonas()` calls return seeded defaults (losing pre-upgrade personas) until the process is recycled. In practice IME processes restart often enough that the next App.onCreate retries. Could add explicit retry in Phase 12 if real-world reports surface.
  - All Phase 2 / 2.5 carry-overs unchanged: localized HeliBoard brand strings in ~10 languages, gesture lib not redistributable, lint baseline with 64 inherited entries, mipmap pruning.
