## Phase 5b Review (carried into Phase 6 prompt)

- **Orchestrator surface:** `TermuxOrchestrator` (singleton via `getInstance(applicationContext)`) exposes `detectStatus()`, `runSetupScript(args, foreground, timeoutMs)`, `reauthProvider(provider)`, `stopBridge()`, `observeBridgeHealth(maxSeconds)`, `fetchProviders()`, `launchFDroidTermuxPage()`, `launchTermuxApp()`. Internal nested types: `Status`, `Provider(cliName)` (separate from `client.Provider` since CLI names differ from API-storage keys), `RunResult`, `ProviderStatus`, `HealthState{Polling|Healthy|Unreachable}`. RUN_COMMAND ↔ suspend bridge: each call generates a UUID, registers a `ConcurrentHashMap<String, CancellableContinuation<RunResult>>` entry, fires the intent via `startForegroundService` with a mutable `PendingIntent.getBroadcast(...)`, and `suspendCancellableCoroutine` until the application-scoped `BroadcastReceiver` resumes it. `withTimeoutOrNull` wraps the await; on timeout / dispatch failure the result carries a synthetic exit code (`EXIT_TIMEOUT=-1`, `EXIT_NOT_PERMITTED=-2`, `EXIT_DISPATCH_FAILED=-3`) so callers don't need separate error paths. Polling uses a dedicated short-timeout (2s connect/socket) Ktor `HttpClient` distinct from `RemoteApiBackend.httpClient` — the latter's 60s socket timeout is sized for SSE streams, not loopback liveness probes.

- **Wizard navigation:** new sub-graph under `AiSettingsRoutes.BACKENDS_TERMUX = "backends/termux"`. `TermuxBridgeRoute` is the entry — calls `detectStatus()` in a `LaunchedEffect(rechecks)` and dispatches: `BRIDGE_RUNNING` → `TermuxBridgeStatusScreen`, anything else → `TermuxSetupWizardScreen`. The wizard renders one of three sub-steps based on `Status`: install Termux (F-Droid deep link + "already installed" recheck), deploy + start bridge (bootstrap command code-block + copy/open-Termux/start buttons + observable health polling that triggers `onRecheck()` on `Healthy`), or done (transitional). Status screen drives re-auth (per provider), restart (full setup re-run), and stop (`bash -c "sv-disable ... && sv down ..."`) via the orchestrator; uptime/auth state refreshes via a refresh counter that re-keys the polling effect after each action completes.

- **Manifest changes:** moved `<uses-permission android:name="com.termux.permission.RUN_COMMAND" />` and `<queries><package android:name="com.termux"/></queries>` from `src/debug/AndroidManifest.xml` to `src/main/AndroidManifest.xml`. Debug overlay now contains only the Phase 4 `TermuxValidationActivity` entry.

- **setup.sh extension:** new `--reauth <provider>` flag (and `--reauth=PROVIDER` form) parsed in `parse_args`. When set, `main()` calls `run_reauth_only` after `require_termux` and exits, skipping `set_allow_external_apps`, `install_termux_packages`, `select_providers`, `install_claude_code_if_selected`, `install_gemini_cli_if_selected`, `deploy_bridge`, `register_services`, `start_bridge`, `print_success`. Existing OAuth handling refactored into `run_oauth_for_claude` / `run_oauth_for_gemini` so reauth and the regular flow share the same `run_one_oauth` helper. Validation: only `claude|gemini` accepted; anything else exits 2 with a usage message. No bridge restart is required — each `/chat` spawns a fresh CLI subprocess that re-reads creds. `setup/README.md` gains a "`--reauth <provider>` (limited mode)" section.

- **Built:** `./gradlew assembleFdroidDebug assemblePlayDebug` clean. `./gradlew assembleFdroidRelease assemblePlayRelease` clean. `./gradlew lintFdroidDebug lintPlayDebug` exit 0; `git diff app/app/lint-baseline.xml` empty; the only remaining non-baseline lint warning in our code is the pre-existing Phase 4 `ObsoleteSdkInt` in `TermuxValidationActivity.kt` (untouched in this phase). `apkanalyzer manifest print` confirms `com.termux.permission.RUN_COMMAND` and `<package name="com.termux"/>` present in BOTH `play-debug` and `play-release-unsigned` APKs (and both fdroid variants). `apkanalyzer dex packages` on the play-debug APK shows `TermuxOrchestrator` (+ nested types), `TermuxBridgeRouteKt`, `TermuxSetupWizardScreenKt`, `TermuxBridgeStatusScreenKt`.

- **Smoke test:** **deferred for human reviewer** — Phase 5b is UX-heavy and the prompt itself notes "unit tests can't catch the experience issues". The on-device flows (from-scratch install, re-auth per provider, restart, stop) require a human at the device and are not feasible to drive purely via tooling in this session. Build/lint/manifest/dex invariants are verified above. `setup.sh --reauth <invalid>` was exercised on the dev box (`bash -n` clean; `--help` lists `--reauth` and `require_termux` correctly bails with exit 1 outside Termux).

- **Deviations from Phase 5b prompt:**
  - `Provider` enum lives nested inside `TermuxOrchestrator` (not as a top-level type) to avoid clashing with `com.aikeyboard.app.ai.client.Provider`, which uses `storageKey`/`displayName` rather than the CLI names this phase needs. The two enums are intentionally separate: API-key surface vs Termux-CLI surface.
  - `detectStatus()` is simplified: returns `TERMUX_NOT_INSTALLED` / `BRIDGE_RUNNING` / `BRIDGE_NOT_RUNNING` (the `TERMUX_INSTALLED_NOT_CONFIGURED` and `SCRIPT_NOT_DEPLOYED` enum values exist but are not currently emitted). Distinguishing those would require a synchronous probe that opens Termux's terminal — clunky UX. The wizard's "deploy + start" step shows the bootstrap command and the start-the-bridge button on the same screen, so a user in either sub-state lands on the right UI either way; dispatch errors surface via the script's exit code + stderr panel.
  - `stopBridge()` added beyond the prompt's listed orchestrator methods, so the status screen's "Stop bridge" button has a typed entry point rather than a string-args call. Internal `runCommand` (private) is the shared workhorse.
  - Wizard step 2 + step 3 collapsed into a single screen for `BRIDGE_NOT_RUNNING`: the bootstrap command is shown above a "Start the bridge" button. First-time users use the curl-and-bash command in Termux; returning users (script previously deployed, bridge died) tap "Start the bridge" directly. Polling auto-advances either way.
  - File-level `@file:SuppressLint("SdCardPath")` on `TermuxOrchestrator.kt` because the hardcoded `/data/data/com.termux/...` paths point at Termux's own filesystem (not ours), making the lint check a false positive.
  - Wizard's bootstrap-command code-block uses a placeholder `https://raw.githubusercontent.com/aikeyboard/ai-keyboard/main/setup/setup.sh` URL — Phase 12 wires the real release asset; for now `setup/README.md` already documents the `adb push setup/setup.sh /data/local/tmp/` dev workflow as the fallback.
