## Phase 7b Review (carried into Phase 8 prompt)

- **Wiring shape:** `CommandRowController.onReadRespondTap` walks first via `A11yProxy.requestScreenContext()`, then dispatches the five `ScreenReaderResult` branches (Success → `handleSuccessfulWalk`; four typed Failures → localized toasts). Walking before consent or backend checks means a user on a screen with no content above the cursor doesn't get a consent prompt for nothing. `handleSuccessfulWalk` short-circuits on blank `aboveInputText`, on missing consent (launches `ReadRespondConsentActivity`), and on missing backend (existing `ai_rewrite_no_backend` toast), then builds the prompt via `ReadRespondPromptBuilder.build(...)` and streams through the existing preview-strip + `BackendResolver.resolve(storage)` pipeline. Same `Dispatchers.Main` coroutine + `streamJob` lifecycle as `onRewriteTap`; `cancel-on-typing` already covers it. `usedSelectionRange = null` ensures commit-time replaces the whole field (Read & Respond is a fresh response, not an in-place rewrite).

- **Flavor split via proxy:** `A11yProxy.kt` lives in `src/fdroid/java/com/aikeyboard/app/ai/a11y/` and `src/play/java/com/aikeyboard/app/ai/a11y/` — never `src/main/`. The fdroid impl calls `ScreenReaderService.instance` directly; the play impl returns `BUILD_DOES_NOT_SUPPORT`. AGP picks one per variant. `BuildConfig.ENABLE_A11Y` is GONE from `CommandRowController` (no longer imported); the proxy carries the entire flavor split. Kept under R8 via a new `proguard-rules.pro` keep rule alongside `BackendResolver`'s.

- **Consent flag:** new `SecureData.readRespondConsented: Boolean = false` (`kotlinx.serialization` default-value back-compat — no schema bump). `SecureStorage.isReadRespondConsented()` / `setReadRespondConsented()` mirror the `selectedBackendStrategy` accessor pattern. `ReadRespondConsentActivity` (Compose, `ComponentActivity`, `platformActivityTheme`, `launchMode="singleTop"`, `exported="false"`, registered in `src/main/AndroidManifest.xml`) shows the title + body + Allow/Decline. Allow flips the flag and finishes; Decline finishes without persisting (next tap re-prompts). Two-tap UX on first use: tap → consent → tap again to fire — acceptable per Open question 1.

- **Privacy invariant:** all four new `Log.w` calls in `CommandRowController` log only `t.javaClass.simpleName` / `it.javaClass.simpleName`. `aboveInputText`, `focusedInputText`, `nodes`, `input`, and `t.message` are NEVER passed to `Log.*`. `ReadRespondPromptBuilder` does no logging at all. **Incidental fix:** `onRewriteTap`'s catch block previously surfaced `t.message` ("Unexpected error: …") to the preview strip — replaced with `ime.getString(R.string.ai_rewrite_stream_failed)` + `Log.w(TAG, "Rewrite stream failed: ${t.javaClass.simpleName}")`. Same root-cause as Phase 6's `RemoteApiBackend` catch-order incidental fix.

- **Open questions for human reviewer:**
  1. **Two-tap UX on first use** — user taps Read & Respond → consent → tap again to actually fire. Acceptable, or worth deep-linking back into the IME via a callback? My read: acceptable. Cross-process callback from an Activity → IME is non-trivial; one-time friction is fine for a one-time consent. (Mirrored from Phase 7b prompt's open question.)
  2. **No deep-link to specific service in `Settings.ACTION_ACCESSIBILITY_SETTINGS`** — user lands on the main Accessibility list and scrolls. OEM-specific deep-link extras exist on some Pixels (`Settings.EXTRA_ACCESSIBILITY_SERVICE_COMPONENT_NAME`) but are not stable. Pursue in Phase 12 polish, or accept here?
  3. **`focusedInputText` redundant with IME's input-connection compose buffer** — Phase 7b uses the value from the screen walk, not from `ime.currentInputConnection`. Both should match, but is there a corner case (composing text mid-prediction) where they diverge? My read: ship the screen-walk value and revisit only if smoke testing shows divergence.
  4. **`readRespondConsented` is a single boolean, not per-backend** — switching from a cloud backend to a Termux backend doesn't re-prompt. Acceptable, or should consent reset when `selectedBackendStrategy` changes? My read: a single flag is fine because the consent body explicitly mentions "the AI backend you selected in Settings → Backends" — the user has full visibility into where data is going. Per-backend consent is over-engineering.
  5. **`ReadRespondPromptBuilder` keep rule** — added to satisfy the §9 invariant table; added beyond what §8.5 mandated. Acceptable, or update PHASE_7b_PROMPT.md's invariant table to drop the "must contain ReadRespondPromptBuilder" row?
