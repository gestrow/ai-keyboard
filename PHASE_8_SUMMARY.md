## Phase 8 Review (carried into Phase 9 prompt)

- **Always-On surface (fdroid only):** new `AlwaysOnService` (fdroid) extends `Service` with `foregroundServiceType="specialUse"` and `PROPERTY_SPECIAL_USE_FGS_SUBTYPE = "screen_reader_companion"`, posts the persistent FGS chip from `ReadRespondNotificationBuilder.buildPersistentChip` on every `onStartCommand`. `ACTION_READ_RESPOND` triggers a single screen walk via `A11yProxy.requestScreenContext()`, runs the same `ReadRespondPromptBuilder.build(...)` + `BackendResolver.resolve(storage)` pipeline as the IME path, and streams deltas/Done/Error into `NOTIF_ID_RESULT` via `postStreaming`/`postCompleted`/`postFailureChip`. `triggerReadRespond` cancels any in-flight job before starting a new one; `START_STICKY` so the chip re-posts after memory-pressure kills. The five `R_FAILURE_*` sentinels surface every failure mode (no-window / generic / no-context / no-backend / stream) as a localized notification body.

- **Tile + boot:** `ReadRespondTileService` (fdroid) maps `qsTile.state` to `storage.isAlwaysOnEnabled()` in `onStartListening`. Tap dispatches `ACTION_READ_RESPOND` to `AlwaysOnService` when always-on is on, else opens `AiSettingsActivity` with `EXTRA_DEEP_LINK_ROUTE = "always-on"`. **API 34+ correction to the prompt:** `startActivityAndCollapse(Intent)` now throws `UnsupportedOperationException` on API 34+ (lint surfaces this), so a `startActivityAndCollapseCompat` helper branches on `SDK_INT` — `PendingIntent` overload on ≥34, deprecated `Intent` overload on 29-33. `BootReceiver` (`directBootAware="false"`) re-starts the service via `AlwaysOnProxy` when `isAlwaysOnEnabled()` after `BOOT_COMPLETED`, wrapped in `runCatching`. `ClipboardCopyReceiver` handles both Copy (with `EXTRA_TEXT`) and Dismiss (no extra) — single receiver, single intent action.

- **Flavor proxy:** `AlwaysOnProxy` lives in `src/fdroid/` and `src/play/`, never `src/main/`. The fdroid impl calls `startForegroundService`/`stopService`; the play impl is a no-op with `isSupported() = false`. Same precedent as Phase 7b's `A11yProxy`. Settings UI gates the toggle on `isSupported()` BEFORE the storage write, so the play APK can never persist `alwaysOnEnabled = true`.

- **Notification builder (src/main, both flavors):** `ReadRespondNotificationBuilder` hosts `NOTIF_ID_*` and `R_FAILURE_*` constants (so `ClipboardCopyReceiver` can reference them without importing `AlwaysOnService`), `createChannels` (LOW for persistent, HIGH for results, idempotent), `buildPersistentChip` (with `Read & Respond now` action via `PendingIntent.getService`), `postStreaming` (BigTextStyle, `setOnlyAlertOnce(true)`, `setOngoing(true)`), `postCompleted` (Copy + Dismiss actions, `setAutoCancel(true)`), `postFailureChip`, `postConsentRequired`, `postServiceNotEnabled`. All cross-flavor intents use `Intent.setClassName(packageName, "<FQN>")` with hardcoded string literals — never `::class.java`. All notifications are `VISIBILITY_PRIVATE` with a redacted `setPublicVersion`. `failureKindToResId` is extracted as JVM-pure helper so unit tests exercise it without a Context.

- **Settings UI (src/main, Compose):** `AlwaysOnRoute` + `AlwaysOnScreen` under `com.aikeyboard.app.ai.ui.alwayson`. Three-gate flow: a11y enabled → consent → POST_NOTIFICATIONS (Android 13+). `DisposableEffect(lifecycleOwner)` + `resumeCount` re-runs gate checks on every ON_RESUME (precedent: `BackendsScreen.kt:60-75`). `notifPermissionLauncher` flips the toggle on once granted. `isAccessibilityServiceEnabled` queries `Settings.Secure.ENABLED_ACCESSIBILITY_SERVICES` with a string-FQN component check (no fdroid-only class import). `AiSettingsNavHost` signature gained `startDestination: String = AiSettingsRoutes.HUB`; `AiSettingsActivity` reads `EXTRA_DEEP_LINK_ROUTE` from intent and threads it through. - **Deviations from Phase 8 prompt:**
  - **`startActivityAndCollapse(Intent)` is not safe on the supported API range as the prompt claimed.** Lint's `StartActivityAndCollapseDeprecated` check flags it as a runtime crash on apps targeting API 34+ (we target 36). Fix: SDK_INT-branched `startActivityAndCollapseCompat` helper that uses `PendingIntent.getActivity` on API 34+ and the Intent overload on 29-33. Both lint suppressions (`@SuppressLint("StartActivityAndCollapseDeprecated")` + `@Suppress("DEPRECATION")`) cover the inner branches.
  - **Added a `Dismiss` action button on `postCompleted`** beyond the `Copy`-only path the prompt's example sketched. The `ai_read_respond_action_dismiss` string was already in the §10 string list, but the example code didn't wire it; lint flagged the string as unused. Reused `ClipboardCopyReceiver` (no `EXTRA_TEXT` → dismiss only) rather than introducing a sibling receiver class.
  - **`ai_always_on_tile_label` moved to `src/fdroid/res/values/a11y_strings.xml`** — only the fdroid manifest references it (the tile service doesn't exist in the play APK), so leaving the string in `src/main/` produced an `UnusedResources` lint error in the play flavor.
  - **`notifySafely` extracted as a private helper** with `@SuppressLint("MissingPermission", "NotificationPermission")` and a `try/catch SecurityException`. The prompt's example called `NotificationManagerCompat.notify(...)` directly; lint flagged five `MissingPermission` errors in the play flavor (where POST_NOTIFICATIONS isn't declared) and a `NotificationPermission` error in both. The helper centralizes the suppression with a brief defensive catch — degrades to silent failure rather than crashing the FGS if permission is revoked mid-session.
  - **`AlwaysOnService.onStartCommand` carries `@SuppressLint("InlinedApi")`** for `FOREGROUND_SERVICE_TYPE_SPECIAL_USE` (API 34 constant). minSdk is 29; older API levels treat the bitmask value as a benign int. Same workaround the prompt's comment described, just with the explicit suppression added.
  - **`onCompletion` removed from the streaming pipeline** — the prompt's example used `.onCompletion { /* notif state already finalized */ }` but the comment itself acknowledged it was a no-op. Dropped to keep the call chain tight.

- **Open questions for human reviewer (Phase 8 spec §"Open questions to resolve at end of phase"):**
  1. **Notification update throttling** — Phase 8 ships without delta-buffering (relies on `setOnlyAlertOnce` + Android's per-channel coalescing). Acceptable, or did smoke testing show jank that needs Phase 12 polish? My read: defer until smoke testing surfaces a problem.
  2. **Tile state when always-on is off** — current spec: tile renders STATE_INACTIVE; tap opens Settings. Alternative: hide tile entirely. Kept STATE_INACTIVE — discoverability wins for a v1 feature.
  3. **Boot receiver and direct-boot** — `directBootAware="false"` defers until user unlock, sidestepping Tink's CE-only Keystore. The few-second delay should be invisible to users; if reports surface, Phase 12 can add a `LOCKED_BOOT_COMPLETED` retry path with a separate DE-storage fallback.
  4. **`POST_NOTIFICATIONS` denial UX** — current spec doesn't deep-link to App Settings → Permissions if the system suppresses re-prompt. The status row's "Grant" chip re-fires `RequestPermission`; the second denial (which the system caches) leaves the user stuck. My read: defer to Phase 12 polish unless smoke testing reveals friction.
  5. **Persistent FGS chip body copy** — kept the action-oriented framing ("Tap a chat's input field, then tap the tile to compose a response"). The privacy-oriented alternative ("Read & Respond is on standby. It runs only when you trigger it.") would communicate what the chip exists for, but the action framing tells the user what to actually do. Open to switching in Phase 12 polish.
