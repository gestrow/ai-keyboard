## Phase 9a Review (carried into Phase 9b prompt)

- **Sticker engine surface (`com.aikeyboard.app.ai.sticker.*`):** `StickerManifest`/`StickerPack`/`Sticker` `@Serializable` data classes (schemaVersion=1, optional `emojis` defaults to empty for forward-compat with Phase 9b). `StickerStorage` is a process singleton over `${filesDir}/stickers/` — `manifest.json` (atomic write via tmp + rename + copy fallback) plus `packs/<pack-id>/sticker_<id>.webp` files. Mutations route through a `synchronized(this) { transform; persist; cached = next; _changes.value++ }` envelope so the in-memory cache and the on-disk JSON never diverge. `changes: StateFlow<Long>` increments on every mutation; Compose routes `collectAsState` it and re-derive their pack lists via `produceState`, so cross-screen edits propagate without manual refresh ticks. `StickerNormalizer` decodes via `ImageDecoder` (API 28+, unconditional on minSdk 29), center-crops to 512×512, and binary-searches WebP quality (90 → 30) for the largest output that fits 100KB. The quality search is extracted as a pure `compressToTargetBytes(measure, budget, min, max)` helper so the loop is testable with a synthetic byte-size curve. `StickerCommitter` wraps `InputConnectionCompat.commitContent` with a `Result { OK, NO_CONNECTION, UNSUPPORTED_FIELD, FAILED }` dispatch; the `acceptsWebp(Array<String>?)` helper is pure-JVM (deviation: see below). `StickerFileProvider` is a 1-line `FileProvider` subclass; manifest authority is `${applicationId}.stickers` (distinct from gesture-data's `${applicationId}.provider`).

- **Keyboard-surface picker (`com.aikeyboard.app.ai.sticker.picker.*`):** `StickerPickerView` is a `FrameLayout` that inflates a 48dp toolbar (back arrow + import button) above a 4-column `RecyclerView` grid. Visibility-flipped against `KeyboardWrapperView` in `main_keyboard_frame.xml` — picker shown ⇒ keyboard wrapper GONE, vice versa. The picker occupies the same vertical region as the keys (`minHeight=240dp` so the IME doesn't shrink while the grid is empty), so `LatinIME.onComputeInsets` does NOT need to change. Colors come from `Settings.getValues().mColors` (`STRIP_BACKGROUND`, `TOOL_BAR_KEY`, `KEY_TEXT`) per the Phase 2.5 keyboard-surface invariant. `StickerGridAdapter` uses `notifyDataSetChanged` (small grids, DiffUtil deferred to Phase 12) and decodes WebPs on the binder thread; ≤100KB inputs decode sub-5ms each, fine for a single-tap picker open.

- **Settings UI (`com.aikeyboard.app.ai.ui.stickers.*`):** Hub-style nav extended with a fifth row (`ai_settings_hub_stickers_*`) routing to `STICKERS_LIST`. `StickerPacksRoute` hoists state via `storage.changes.collectAsState() + produceState`, hosts the `rememberLauncherForActivityResult(PickMultipleVisualMedia(30))` photo picker, and routes new imports through `StickerNormalizer` on `Dispatchers.IO` then `storage.addSticker` on the originating coroutine scope. `StickerPacksScreen` renders the empty-state when no pack contains stickers (button → import); else a `LazyColumn` of pack rows with `pluralStickerCount`. `StickerPackEditRoute`+Screen mirrors `PersonaEditScreen` patterns: TextField for pack name (Save button enabled iff trimmed name differs and is non-empty), 4-column `LazyVerticalGrid` of sticker thumbnails with long-press → `AlertDialog` delete confirmation, separate "Delete pack" affordance with its own confirmation dialog, top-bar action button reuses the launcher to import more. Each thumbnail loads its bitmap on `Dispatchers.IO` via `LaunchedEffect(sticker.id, file.absolutePath)` to keep composition off the binder thread.

- **CommandRowController integration:** `setStickerPicker(view)` caches the picker reference and the sibling `keyboard_view_wrapper` (resolved once via `(view.parent as ViewGroup).findViewById`). Picker callbacks dispatch to `commitSticker(packId, stickerId)` (success → `hidePicker()`; the four `Result` branches map 1:1 to localized toasts), `launchSettingsAtStickers()` (deep-link via `EXTRA_DEEP_LINK_ROUTE = STICKERS_LIST` — Phase 8 plumbing), and `hidePicker()`. New `onInputStarted(EditorInfo?)` is invoked from `LatinIME.onStartInputViewInternal` immediately after `super.onStartInputView` so the picker is dismissed on every input transition (including `editorInfo == null`); the `EditorInfo` is also cached so `commitSticker` has a stable handle even if `getCurrentInputEditorInfo()` momentarily returns null mid-transition. Authority is constructed at call time as `"${ime.packageName}.stickers"` (matches the manifest placeholder for both flavors via `applicationIdSuffix`).

- **Deviations from Phase 9a prompt:**
  - **No `READ_MEDIA_IMAGES` permission added.** ARCHITECTURE.md's Phase 9 row mentioned it but the prompt itself ratified the deviation (§"Architectural decisions ... 1"): AndroidX's `PickMultipleVisualMedia` returns content URIs through the system photo picker on API 33+ and falls back to SAF on 29–32, both of which grant per-URI read access without a manifest permission. Documented here per the prompt's instruction.
  - **`StickerCommitter.acceptsWebp` rewritten as a pure-JVM helper** instead of taking `EditorInfo` directly. The prompt's example used `ClipDescription.compareMimeTypes` inside the helper, which requires Android — incompatible with the prompt's "None require Robolectric" mandate for the test file. Replaced with explicit lowercase-trim + literal match against `image/webp` / `image/*` / `*/*`. Same wildcard semantics as `compareMimeTypes` for the subset we care about; unit-testable on plain JVM. The original Android-dependent entry point survives as a private wrapper inside `StickerCommitter.insert` for the production call path.
  - **Layout import button uses `ic_ai_settings_gear`** as the prompt predicted (§8 explicitly allowed this placeholder); the content-description carries the semantic "Open Settings to import stickers" so screen readers don't mislead. Phase 12 polish can swap in a dedicated "+" icon if the gear UX-tests poorly.
  - **`?attr/selectableItemBackgroundBorderless` → `?android:attr/selectableItemBackgroundBorderless`.** HeliBoard does not depend on AppCompat (Phase 2 deviation), so the AppCompat theme attr resolves to no resource at link time. Switched to the framework attr; visually identical.
  - **`StickerStorage.kt` uses `com.aikeyboard.app.latin.utils.Log` (HeliBoard's wrapper) instead of `android.util.Log` directly**, and wraps the corrupt-manifest `Log.w` call in `runCatching {}`. Without the wrap, the `getManifest_handlesCorruptManifestGracefully` test crashes on the JVM Stub! exception (the project doesn't enable `unitTests.isReturnDefaultValues`). The corrupt-manifest path is recoverable (we proceed with an empty manifest in memory; the corrupt file is preserved per Phase 3a's data-preserving precedent), so silent log failure is acceptable.
  - **Added a Proguard keep rule for `StickerCommitter`** (`-keep class com.aikeyboard.app.ai.sticker.StickerCommitter { public static *** insert(...); static *** acceptsWebp(...); }`). Without it, R8 inlined the singleton's `insert` method (single call site in `CommandRowController.commitSticker`) and the class disappeared from both release dexes — same precedent as Phase 6's `BackendResolver` and Phase 7b's `ReadRespondPromptBuilder`. Required to satisfy §17's dex invariant table.
  - **Picker layout uses `minHeight="240dp"`** so the visibility flip against `KeyboardWrapperView` doesn't shrink the IME window. The prompt's `wrap_content` height alone left the picker at toolbar+empty-message height (~120dp), which collapsed the IME below the keyboard's normal height when shown empty. Visually-stable swap rules.
  - **Added `androidx.activity:activity-compose` as an explicit dependency** (BOM-managed, no version pin) per the prompt's §2 instruction. The dependency was already transitive via `navigation-compose`; making it direct insulates Phase 9a from a future navigation-compose bump that drops the transitive.

- **Open questions for human reviewer (carry into 9b prompt):**
  1. **`StickerCommitter.acceptsWebp` deviation** — the pure-JVM matcher accepts `image/webp` / `image/*` / `*/*`. ClipDescription's full algorithm also matches `*/webp` (rare edge case). Acceptable to keep the simplified set, or worth adding the `subtype-only` wildcard branch for parity?
  2. **`StickerStorage` `Log.w` wrapped in `runCatching`** — privacy-clean (only logs `javaClass.simpleName`), but the wrap is purely a unit-test workaround. Cleaner alternative: enable `unitTests.isReturnDefaultValues = true` project-wide. My read: the wrap is fine for v1 — the corrupt-manifest path is the only Log site in `StickerStorage`, and the project hasn't needed `isReturnDefaultValues` for any other test until now.
  3. **Multi-pack tabs on the keyboard picker** — Phase 9a ships with a flat grid. Once 9b lands the WhatsApp pack-identity contract, packs become first-class. My read: yes to tabs in 9b (each WhatsApp pack is a deliberate organizational unit), but defer tab UX polish to Phase 12.
  4. **Sticker thumbnail caching** — `StickerGridAdapter.onBindViewHolder` decodes from disk on the binder thread. Fine for ≤30 stickers; defer LRU/Coil/Glide to Phase 12 unless smoke testing surfaces hitches.
  5. **Pack-size soft cap** — `MAX_IMPORTS_PER_BATCH = 30` caps a single import batch but not total stickers in a pack. After multiple batches a pack could hold 100+ stickers, all of which the picker decodes synchronously into ~1MB ARGB_8888 bitmaps each. Phase 12 task to add a per-pack cap (e.g., 50) plus a one-line `BitmapPool` if smoke testing surfaces OOM.
  6. **Picker dismissal on hardware back press** — Phase 9a doesn't wire `onKeyDown(KEYCODE_BACK)`; users dismiss via the picker's own back arrow. Acceptable for v1; Phase 12 can wire it if the no-back-button UX feels off.
