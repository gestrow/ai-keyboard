## Phase 9b Review (carried into Phase 10 prompt)

- **WhatsApp sticker pack contract (`com.aikeyboard.app.ai.sticker.WhatsAppStickerContentProvider`):** standalone `ContentProvider` with authority `${applicationId}.whatsapp.stickers` (distinct from 9a's `StickerFileProvider` `${applicationId}.stickers` and from the legacy gesture authority). URI hierarchy `metadata`, `metadata/<pack-id>`, `stickers/<pack-id>`, `stickers_asset/<pack-id>/<filename>` matches WhatsApp's spec; 15-column metadata cursor + 3-column sticker cursor expose the pack manifest from `StickerStorage` + emoji CSV (comma-joined) + accessibility text (space-joined, falls back to `pack.name`). All write methods (`insert`/`update`/`delete`) are no-ops. **Privacy gate:** `assertAllowedCaller()` rejects any caller whose `getCallingPackage()` is not in `{com.whatsapp, com.whatsapp.w4b}` with `SecurityException`; gate is on `query` + `openAssetFile` only — `getType()` is intentionally ungated because the framework calls it from our own process (callingPackage == null) for intent-resolution. Path traversal is rejected in `openAssetFile` (path components containing `/` or `..` return null). `metadataRow` and `stickerRows` are JVM-pure helpers extracted as `internal` `@JvmStatic` for unit testing without Robolectric.

- **Tray-icon import path (`TrayIconNormalizer`):** mirrors 9a's `StickerNormalizer` but produces 96×96 PNG ≤50KB. PNG quality is lossless, so the byte-budget search downsamples bitmap dimensions in 8px steps from 96 → 32 instead of walking lossy quality. Fast path returns the first attempt (96×96 rarely overflows 50KB). `encodeFitting(compress, targetPx, budget)` overload extracts the search loop as a JVM-pure helper. Storage gains `setTrayIcon(packId, fileName)` + `trayIconFile(packId, fileName)` sibling to `stickerFile`.

- **Pack-edit affordances (`StickerPackEditScreen` extensions):** four new affordances stacked into the existing screen — tray-icon row (current bitmap thumbnail + "Choose / Replace tray icon" button → single-image `PickVisualMedia` → `TrayIconNormalizer`), publisher `OutlinedTextField` (deferred-save on focus loss to avoid bumping `imageDataVersion` per keystroke), per-sticker emoji `OutlinedTextField` below each thumbnail (same deferred-save pattern, `parseEmojis` takes first 3 whitespace-separated tokens), preflight chip column rendering one `AssistChip` per `StickerPackValidator.Issue`, and an "Add to WhatsApp" `Button` enabled iff `issues.isEmpty()` and `whatsAppStatus != NONE`. `whatsAppStatus` is plumbed from the route so the screen stays Compose-pure. The thumbnail grid switched from 4 columns to 3 to make room for the emoji TextField below each cell.

- **`StickerPackValidator`:** sealed `Issue` enum with seven values (`TOO_FEW_STICKERS`, `TOO_MANY_STICKERS`, `MISSING_TRAY_ICON`, `MISSING_PUBLISHER`, `NAME_TOO_LONG` 128, `PUBLISHER_TOO_LONG` 128, `IDENTIFIER_INVALID` ASCII `[A-Za-z0-9_.-]+`). `validate()` returns ALL failing issues so the UI renders chips simultaneously, not one-at-a-time error chasing. UUIDs from `StickerStorage.createPack` (lowercase hex with hyphens) pass the identifier regex.

- **`AddToWhatsAppHelper`:** detection (`status(context)` → `CONSUMER_ONLY`/`BUSINESS_ONLY`/`BOTH`/`NONE`) and intent assembly (`buildIntent(packId, authority, packName)`). The intent fires `com.whatsapp.intent.action.ENABLE_STICKER_PACK` *without* a target package — Android's chooser appears when both apps are installed. `statusFromInstalled(installed: List<String>)` is the JVM-pure helper for unit tests; the only Android dep (`getPackageInfo`) is isolated in `isInstalled`.

- **On-keyboard pack tabs (`StickerPackTabsView` + integration):** horizontal `RecyclerView` of 48dp `ImageButton` tabs (64dp total height with vertical padding) sits between the picker's toolbar and the grid in `sticker_picker.xml`. Each tab shows the pack's tray icon (or `ic_sticker` fallback for tray-less packs); selected tab renders at alpha 1.0, others at 0.5. `StickerPackTabsView` is a static-nested adapter (NOT `inner`) so the IME's view recreation on theme changes leaves the adapter GC-eligible. `StickerPickerView.bind` caches the last-bound `(packs, resolver)` so tab taps round-trip locally without going through `CommandRowController`. Default-selected pack: first pack with stickers, else first pack. Stale-selection guard: if the previously selected pack is deleted while the picker is open, the cached id resets so the default-selection fallback runs. `StickerGridAdapter.update` gained an optional `selectedPackId: String? = null` parameter; non-null filters items to that pack only. The picker pre-filters before calling `update` (the param is informational/defensive), so `notifyDataSetChanged` runs on the smaller list.

- **9a carry-over #1 — sticker-context error chip:** new `StickerPickerView.showError(@StringRes)` flips an inline `TextView` (`@id/sticker_picker_error_chip`, top of the picker layout above the toolbar) visible for 2.5s with the localized string, then auto-fades via a `Handler` token (`postDelayed` + `removeCallbacks`-on-replace to avoid stale messages). `CommandRowController.commitSticker` swaps all four `toast(...)` calls (`NO_CONNECTION` / `UNSUPPORTED_FIELD` / `FAILED` / `missing_file`) for `stickerPicker?.showError(...)`. The chip uses HeliBoard's `ColorType.ACTION_KEY_BACKGROUND` (the "raised attention" slot per `Colors.kt`) with `KEY_TEXT` foreground — runtime-color invariant from Phase 2.5 preserved.

- **9a carry-over #2 — post-pack-delete blank screen:** `StickerPackEditRoute` gained `LaunchedEffect(pack) { if (pack == null) onBack() }` placed AFTER the synchronous initial-value lookup in `produceState`. `pack` is non-null on first composition for any existing `packId`, so the LaunchedEffect fires only after a real delete. `onDeletePack` no longer calls `onBack()` directly — it just calls `storage.deletePack(packSnapshot.id)` and lets the StateFlow recomposition trigger the LaunchedEffect-driven pop, eliminating the one-frame window where a null pack would render a blank surface.

- **`StickerStorage.update()` envelope:** existing `transform → persist → cached/_changes` envelope extended to compare each pack pre/post via `packsContentEqual(name, publisher, trayIconFile, avoidCache, stickers)` and bump `imageDataVersion` for any pack whose content-bearing tuple changed. `imageDataVersion` itself + `createdAt` are excluded from equality (otherwise the bump would cascade indefinitely). WhatsApp re-fetches a pack whenever this differs from its cached value; the version string is a `Long.toString` of a monotonic counter — `Long.MAX_VALUE` is unreachable at any human edit cadence.

- **Storage extensions:** `setTrayIcon(packId, fileName)`, `setPublisher(packId, publisher)`, `setStickerEmojis(packId, stickerId, emojis: List<String>)`, plus the `trayIconFile(packId, fileName): File` resolver. All four go through the standard `update(transform)` envelope so `imageDataVersion` bumps and `_changes.value++` propagation ride along automatically.

- **Data model (`StickerModels.kt`):** `StickerPack` gained `trayIconFile: String? = null`, `publisher: String = ""`, `imageDataVersion: Long = 1L`, `avoidCache: Boolean = false`. All four are default-valued so 9a manifest blobs (no schema bump) decode without migration. `Sticker.emojis` was already in place from 9a — Phase 9b's emoji editor and the cursor rows consume it directly.

- **Manifest changes (src/main/):** new `<provider>` for `WhatsAppStickerContentProvider` with `android:exported="true"` (WhatsApp must reach it from its own process) and `android:grantUriPermissions="false"` (we serve files via `openAssetFile` not URI grants); `tools:ignore="ExportedContentProvider"` justified inline (the privacy gate is in code, not at the manifest level — Android does not allow signature-based gates against arbitrary cross-vendor packages). Two new `<package>` entries inside the existing `<queries>` block (`com.whatsapp`, `com.whatsapp.w4b`) so `getPackageInfo` / `queryIntentActivities` return non-empty for these packages on Android 11+. **No new `<uses-permission>` entries** — permission set unchanged from Phase 8.

- **Deviations from Phase 9b prompt:**
  - **Sticker grid columns reduced from 4 to 3 in pack-edit screen** to fit the per-sticker emoji `OutlinedTextField` cleanly under each thumbnail. The picker grid stays at 4 columns (no emoji TextField below picker thumbnails). The prompt's §10c sketch didn't specify column count; this is a layout polish judgement.
  - **`onDeletePack` no longer calls `onBack()` from the route** — `storage.deletePack(...)` alone is enough because the new `LaunchedEffect(pack)` pops back automatically when the StateFlow recomposition observes pack-disappearance. Calling `onBack()` directly here would race with the LaunchedEffect (the same race that produced the carry-over #2 bug in 9a). Cleaner to use a single dispatch path.
  - **`StickerPackTabsView`'s static-nested `TabAdapter` constructor accepts the click callback directly** (per the prompt's §9 sketch). The first cut had two `tabAdapter` declarations in `init {}` (the prompt's example was duplicated); collapsed to one — initialized at the property declaration site so the `init` block can register it as the RecyclerView adapter.
  - **`tools:ignore="Overdraw,ContentDescription"` on `sticker_pack_tab.xml`** — lint flagged both. Overdraw is intentional (parent paints `STRIP_BACKGROUND`, each tab paints the framework ripple drawable for press feedback); ContentDescription is set programmatically by the adapter (per pack name). Inline justifications in the layout file.
  - **`tools:ignore="ExportedContentProvider"` on the `<provider>` element** — `exported="true"` is required because WhatsApp queries from its own process; the privacy gate is the in-code `assertAllowedCaller()` allowlist. Android's signature-based permission gate is not an option for cross-vendor IPC against unstable WhatsApp signing certs.
  - **`StickerPackTabsView` uses a single `tabAdapter` constructor invocation** — in the prompt's §9 the TabAdapter is declared twice (once with no callback, once with the callback). Collapsed to one. Pure prompt-error fix; behavior identical.
  - **Dropped four prompt-listed strings as unused** — `ai_stickers_emoji_label`, `ai_stickers_whatsapp_added`, `ai_stickers_whatsapp_failed`, `ai_stickers_picker_tab_desc`. The first was redundant with `ai_stickers_emoji_hint` (which serves as both label and placeholder); `_added` / `_failed` were prompt placeholders for a future success/failure toast on the WhatsApp result callback (v1 leaves the result handler empty per §10d, since WhatsApp shows its own confirmation UI); `_picker_tab_desc` was unused because tab content descriptions use `pack.name` directly. Lint flagged all four as `UnusedResources`; removed to keep the lint-baseline diff empty.
  - **`TrayIconNormalizer.compressAtPx` uses `bitmap.scale(px, px, filter = true)` from `androidx.core.graphics.scale`** instead of `Bitmap.createScaledBitmap`. Lint's `UseKtx` flagged the framework spelling; the core-ktx extension is functionally identical and matches `StickerNormalizer`'s precedent.

- **Open questions for human reviewer (carry into Phase 10 prompt):**
  1. **Calling-package allowlist hardening.** `WhatsAppStickerContentProvider.ALLOWED_CALLERS = {com.whatsapp, com.whatsapp.w4b}`. If WhatsApp ships a third package (region-specific, business v2, etc.) we deny access until we add it. Signature pinning against WhatsApp's known certs is not viable (fragile across regional builds). My read: keep the allowlist tight; users on unknown WhatsApp builds report → we add their package in Phase 12.
  2. **`imageDataVersion` upper bound.** WhatsApp's contract says column type is "String" but doesn't specify whether they parse `Long.MAX_VALUE` correctly. Not a real concern at human edit cadences (~9 quintillion edits per pack); deferred.
  3. **Animated stickers.** ARCHITECTURE.md defers animated WebP to v2. Phase 9b hardcodes `animated_sticker_pack=0` for all packs; updating both the cursor row and `StickerNormalizer`'s decode path is a Phase 11+ task.
  4. **Per-sticker emoji parser fidelity.** `parseEmojis` does `split("\\s+").take(3)`, which doesn't handle ZWJ sequences or skin-tone modifiers if the user types a space inside a multi-codepoint cluster. The user-facing contract is "separate emojis with spaces"; phase 12 polish can swap in a `BreakIterator`-based grapheme-cluster parser if real-world usage shows mismatches.
  5. **Tray icon overrides for tabs.** Pack tabs show the tray icon if set, else fall back to `ic_sticker`. Could render the first sticker's thumbnail as the fallback for visual differentiation; deferred to Phase 12 to avoid extra disk I/O on the keyboard surface during display.
  6. **Multi-pack emoji-tagging UX.** Each sticker's `OutlinedTextField` lives inside its grid cell — at 30 stickers per pack max the screen has up to 30 TextFields. Compose handles it but the screen scrolls a lot; Phase 12 could add a bulk-edit dialog. Defer pending real-world feedback.
  7. **`StickerPackValidator.MISSING_TRAY_ICON` doesn't verify the file exists on disk.** It only checks `pack.trayIconFile.isNullOrEmpty()`. If the manifest references `tray.png` but the file was deleted out-of-band (`adb` shenanigans, app-data wipe edge case), validation passes but WhatsApp's `openAssetFile` returns null and shows a broken icon. Stricter validator would accept a `(File) -> Boolean` checker; deferred to Phase 12 — v1 failure mode is recoverable (user re-picks).
  8. **Phase 8 boot-persistence root cause.** Surfaced in 9a smoke (now documented as a Phase 9b carry-over per the prompt). Investigate whether `BootReceiver` is firing, whether `isAlwaysOnEnabled` returns false during the post-boot Tink-keystore-unlock window, or whether `startForegroundService` is silently failing in `BOOT_COMPLETED` context.
